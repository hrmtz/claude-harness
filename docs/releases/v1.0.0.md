# v1.0.0 public-safe harness

This release makes the repository safe to publish as an open-source harness.
It removes hardcoded personal paths and repo names from runtime behavior, makes
incident automation opt-in, and keeps legacy state paths working where possible.

## Scope

- `harness-core`: credential leak handling, guard rails, and temporal/session context
- `harness-formation`: peer-worker orchestration, mailbox state, and skill naming
- `harness-rails`: long-run anomaly detection and repo mapping
- `scripts/`: backup, red-team cron, and cross-CLI path resolution

## `harness-core`

### Credential leak handling

- `plugins/harness-core/hooks/credential_scrub.py` still sanitizes the active
  session JSONL in place when a credential value is detected.
- Incident issue filing is now opt-in. It only runs when both of these are set:
  - `HARNESS_CREDENTIAL_LEAK_ISSUES=1`
  - `CREDENTIAL_LEAK_ISSUE_REPO=owner/repo`
- The last filed issue reference is stored as `owner/repo#number`, and stale
  references are ignored when the target repo changes.
- `plugins/harness-core/hooks/autorotate_leaked_cred.sh` now requires an audited
  operator runbook via `HARNESS_AUTOROTATE_SCRIPT`.
- Optional notification targets are still supported through:
  - `HARNESS_AUTOROTATE_ISSUE_REPO`
  - `HARNESS_AUTOROTATE_DISCORD_CHANNEL`

### File-read guard

- `plugins/harness-core/hooks/credential_file_read_guard.sh` blocks full-file
  reads of plain credential sources such as `.env`, `.env.*`, `rclone.conf`,
  `.netrc`, `~/.aws/credentials`, Cloudflared tunnel JSON, and private key files.
- Template/test fixtures are exempted:
  - `*.env.example`
  - `*.env.template`
  - `*.env.sample`
  - `*.env.dist`
  - `*.env.test`
  - `*.env.local-example`
- One intentional read is allowed through a one-shot ack file at
  `~/.claude/state/cred_read_ack`.
- The guard now tells the operator to use `grep -c <KEY>` or `cut -d= -f1`
  for structure-only inspection, and `sops exec-env` for actual use.

### Command guards

- `plugins/harness-core/hooks/bash_command_guard.sh` keeps blocking unsafe
  Bash patterns, but the conscious bypasses are now explicit `HARNESS_*` flags.
- `plugins/harness-core/hooks/branch_policy_guard.sh` uses the same explicit
  `HARNESS_ACK_*` style for deliberate main-branch operations.
- `HARNESS_ACK_CRED_READ=1` is now a deliberate bypass for the command guard;
  it is no longer treated as a generic exportable state for file reads.

### Temporal and companion context

- `plugins/harness-core/hooks/temporal_anchor.sh` now derives the project root
  from the hook input `cwd` and emits:
  - current date
  - project root
  - project age in days
  - last commit date
  - last 7 days of commit activity
  - optional recent memory file mtimes
- Optional memory grounding is enabled with `HARNESS_TEMPORAL_MEMORY_DIR`.
- `plugins/harness-core/hooks/ghost_inject.sh` and
  `plugins/harness-core/hooks/recent_topics_inject.sh` now:
  - default to `HARNESS_HIPPOCAMPUS_HOME` or `~/projects/hippocampus-mcp`
  - accept `HARNESS_HIPPOCAMPUS_CREDS_DIR`, `HARNESS_HIPPOCAMPUS_SECRETS`,
    `HARNESS_GHOST_INJECT_SCRIPT`, `HARNESS_RECENT_TOPICS_SCRIPT`,
    `HARNESS_HIPPOCAMPUS_PYTHON`
  - silently no-op when disabled or when the companion files are absent

## `harness-formation`

- `plugins/harness-formation/bin/formation`,
  `plugins/harness-formation/lib/mailbox.sh`,
  `plugins/harness-formation/lib/mailbox_relay.sh`, and
  `plugins/harness-formation/bin/mailbox-retention` now resolve runtime state in
  this order:
  1. explicit `FORMATION_HOME`
  2. explicit `NJSLYR_HOME`
  3. legacy `~/.njslyr7` if `~/.formation` does not exist
  4. default `~/.formation`
- `NJSLYR_HOME` is retained as a legacy alias for callers that still inspect it.
- LINE notification code was removed from the command path; `formation done`
  and `formation ask` are mailbox-only.
- The formation suggestion hook now surfaces the generic `formation skill`
  name instead of the old project-specific name.
- The README and skill docs now describe plugin install plus symlinking, not a
  standalone `install.sh`.

## `harness-rails`

- `plugins/harness-rails/lib/safety_rails/watcher.py` no longer assumes a
  repository mapping for the public project.
- GitHub issue emit is opt-in through either:
  - `HARNESS_RAILS_PROJECT_REPOS='{"project":"owner/repo"}'`
  - `~/.config/safety-rails/project_repos.json`
- When no mapping is configured, the watcher still reports anomalies, but it
  does not create GitHub issues.
- Discord notification remains available through the existing local CLI path.

## `scripts/` and cross-CLI helpers

- `scripts/backup_jsonl_to_r2.sh` now fails closed unless both of these are set:
  - `HARNESS_JSONL_BACKUP_DEST`
  - `HARNESS_JSONL_BACKUP_RECIPIENT`
- `scripts/test_backup_jsonl_config.sh` covers that fail-closed behavior.
- `scripts/lib/cross_cli_externals.py` now recognizes `${HIPPOCAMPUS_HOME}` in
  addition to `${HARNESS_DIR}` and `${HOME}`.
- `scripts/run_red_team_cron.sh`, `scripts/red_team.crontab`, and
  `scripts/red_team_cron_prompt.md` now take repo, Discord, and issue targets
  from env instead of hardcoding author-specific paths.
