# v1.0.0 公開安全化リリース

このリリースで、リポジトリをオープンソースのハーネスとして公開できる形に整えました。
実行時にハードコードされていた個人用パスやリポジトリ名を排除し、事故対応の自動化は明示的な opt-in にし、可能な範囲で旧来の状態パスも維持しています。

## 対象範囲

- `harness-core`: クレデンシャル漏洩対応、ガードレール、時系列/セッション文脈
- `harness-formation`: 共同ワーカーの起動・調停、メールボックス状態、スキル名
- `harness-rails`: 長時間処理の異常検知、GitHub リポジトリ対応
- `scripts/`: バックアップ、red-team cron、クロス CLI のパス解決

## `harness-core`

### クレデンシャル漏洩対応

- `plugins/harness-core/hooks/credential_scrub.py` は、クレデンシャル値を検知すると、動作中の session JSONL をその場で sanitize します。
- issue への自動記録は opt-in です。次の両方が設定されたときだけ動作します。
  - `HARNESS_CREDENTIAL_LEAK_ISSUES=1`
  - `CREDENTIAL_LEAK_ISSUE_REPO=owner/repo`
- 最後に作成した issue 参照は `owner/repo#number` 形式で保存され、対象リポジトリが変わった古い参照は無視されます。
- `plugins/harness-core/hooks/autorotate_leaked_cred.sh` は、監査済みの運用手順を `HARNESS_AUTOROTATE_SCRIPT` で明示する必要があります。
- 追加の通知先は引き続き利用できます。
  - `HARNESS_AUTOROTATE_ISSUE_REPO`
  - `HARNESS_AUTOROTATE_DISCORD_CHANNEL`

### ファイル読みガード

- `plugins/harness-core/hooks/credential_file_read_guard.sh` は、`.env`、`.env.*`、`rclone.conf`、`.netrc`、`~/.aws/credentials`、Cloudflared の tunnel JSON、private key ファイルなどの全文読みをブロックします。
- ただしテンプレートやテスト用 fixture は除外されます。
  - `*.env.example`
  - `*.env.template`
  - `*.env.sample`
  - `*.env.dist`
  - `*.env.test`
  - `*.env.local-example`
- 1 回だけの意図的な読み取りは `~/.claude/state/cred_read_ack` で許可できます。
- ガードは、構造だけを確認するなら `grep -c <KEY>` または `cut -d= -f1` を使うよう案内し、実際の利用には `sops exec-env` を使うよう促します。

### コマンドガード

- `plugins/harness-core/hooks/bash_command_guard.sh` は危険な Bash パターンを引き続きブロックしますが、意図的な bypass は `HARNESS_*` フラグで明示する形に整理しました。
- `plugins/harness-core/hooks/branch_policy_guard.sh` も、main ブランチへの意図的な操作には同じく `HARNESS_ACK_*` 形式を使います。
- `HARNESS_ACK_CRED_READ=1` は command guard に対する明示的な bypass であり、file read の一般的な exportable state としては扱いません。

### 時系列と companion 文脈

- `plugins/harness-core/hooks/temporal_anchor.sh` は、hook 入力の `cwd` から project root を求め、次の情報を出します。
  - 現在日付
  - project root
  - project age の日数
  - 最終 commit 日
  - 直近 7 日の commit 件数
  - 任意の memory file の mtime
- 追加の時系列 grounding は `HARNESS_TEMPORAL_MEMORY_DIR` で有効にできます。
- `plugins/harness-core/hooks/ghost_inject.sh` と `plugins/harness-core/hooks/recent_topics_inject.sh` は次のように変わりました。
  - `HARNESS_HIPPOCAMPUS_HOME` または `~/projects/hippocampus-mcp` を既定にする
  - `HARNESS_HIPPOCAMPUS_CREDS_DIR`、`HARNESS_HIPPOCAMPUS_SECRETS`、`HARNESS_GHOST_INJECT_SCRIPT`、`HARNESS_RECENT_TOPICS_SCRIPT`、`HARNESS_HIPPOCAMPUS_PYTHON` を受け付ける
  - 無効化時や companion ファイル欠如時には静かに no-op する

## `harness-formation`

- `plugins/harness-formation/bin/formation`、
  `plugins/harness-formation/lib/mailbox.sh`、
  `plugins/harness-formation/lib/mailbox_relay.sh`、
  `plugins/harness-formation/bin/mailbox-retention` は、runtime state を次の順で解決します。
  1. 明示的な `FORMATION_HOME`
  2. 明示的な `NJSLYR_HOME`
  3. `~/.formation` が存在しない場合の legacy `~/.njslyr7`
  4. 既定の `~/.formation`
- `NJSLYR_HOME` は、まだ参照している呼び出し元向けの legacy alias として残しています。
- LINE 通知のコードはコマンド経路から削除されました。`formation done` と `formation ask` は mailbox のみを使います。
- formation の suggestion hook は、旧プロジェクト固有名ではなく一般名の `formation skill` を示すようになりました。
- README と skill doc は、単独の `install.sh` ではなく plugin install + symlink の手順を説明します。

## `harness-rails`

- `plugins/harness-rails/lib/safety_rails/watcher.py` は、公開プロジェクトに対して固定の repository mapping を前提にしません。
- GitHub issue の発行は opt-in です。次のどちらかで設定します。
  - `HARNESS_RAILS_PROJECT_REPOS='{"project":"owner/repo"}'`
  - `~/.config/safety-rails/project_repos.json`
- mapping が未設定でも watcher は異常を報告しますが、GitHub issue は作成しません。
- Discord 通知は既存の local CLI 経路でそのまま利用できます。

## `scripts/` と cross-CLI ヘルパー

- `scripts/backup_jsonl_to_r2.sh` は、次の両方が設定されていない限り fail closed します。
  - `HARNESS_JSONL_BACKUP_DEST`
  - `HARNESS_JSONL_BACKUP_RECIPIENT`
- `scripts/test_backup_jsonl_config.sh` は、この fail-closed 動作を検証します。
- `scripts/lib/cross_cli_externals.py` は、`${HARNESS_DIR}` と `${HOME}` に加えて `${HIPPOCAMPUS_HOME}` も認識します。
- `scripts/run_red_team_cron.sh`、`scripts/red_team.crontab`、`scripts/red_team_cron_prompt.md` は、repo・Discord・issue の対象を env から受け取るようになり、作者固有のパスを直書きしません。
